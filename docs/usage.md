# Usage Guide

CLI flags, alternative run modes, governance toggles, per-feature commands, and
test-suite invocations. The README covers the happy path; this file covers
everything else.

## Configuring AI providers

The pipeline reads provider configuration from `.env`. Three setups are supported:

### Anthropic (default)

```dotenv
AI_PROVIDER=anthropic
ANTHROPIC_API_KEY=sk-ant-...
ANTHROPIC_MODEL=claude-haiku-4-5-20251001
```

Get a key at <https://console.anthropic.com/>.

### OpenAI

```dotenv
AI_PROVIDER=openai
OPENAI_API_KEY=sk-...
OPENAI_MODEL=gpt-4.1-mini
```

Get a key at <https://platform.openai.com/api-keys>.

### Any OpenAI-compatible provider

Anything that speaks the OpenAI Chat Completions protocol — third-party gateway, self-hosted endpoint, etc. Use the OpenAI block plus a custom base URL and model name:

```dotenv
AI_PROVIDER=openai
OPENAI_API_KEY=<your-api-key>
OPENAI_MODEL=<provider-model-name>
OPENAI_BASE_URL=<provider-openai-compatible-base-url>
```

### No AI (deterministic fallback)

Leave all keys blank. The pipeline will warn and prompt for confirmation at runtime, or skip the prompt with `--no-ai`:

```bash
python main.py --no-ai
```

Outputs the same JSON contract; only the prose inside `explanation` and `corrective_action` is generated by rule-based fallback rather than an LLM.

## CLI Reference

```bash
# Full orchestrated run
python main.py --mode orchestrated --input data/input/billing.csv --verbose

# Choose an AI provider
python main.py --ai-provider openai --input data/input/billing.csv

# Run without AI provider calls
python main.py --no-ai --input data/input/billing.csv

# Select a different client rule set
python main.py --client client_b --input data/input/billing.csv

# Launch guided terminal menu
python main.py --interactive
```

### API-key confirmation flow

If you run the CLI without setting `ANTHROPIC_API_KEY` (or `OPENAI_API_KEY` when `--ai-provider openai` is selected) and **without** passing `--no-ai`, the CLI will refuse to silently downgrade. Instead, it warns that the run will use the deterministic / programmatic fallback and asks for explicit confirmation:

```
⚠  No API key found for provider: anthropic
   Expected env var: ANTHROPIC_API_KEY
   …
Continue in deterministic mode? [y/N]:
```

Three ways to bypass the prompt:

| Situation | Use |
|-----------|-----|
| You intentionally want the deterministic mode | `--no-ai` |
| You're in CI / non-interactive context | `--yes` (or simply pipe stdin — non-TTY is auto-accepted with a stderr notice) |
| You want AI mode | Set `ANTHROPIC_API_KEY` (or `OPENAI_API_KEY`) and re-run |

The intent is governance-first: the deterministic fallback is fully supported, but it must be a **conscious choice**, not an accidental degradation.

## Reviewer self-guided tour

Each command below exercises one capability. The full tour takes 3–5 minutes if you skip the AI-mode commands, 10 minutes with AI on.

### Run modes

Deterministic-mode run. Full pipeline, no LLM call, same JSON contract as AI mode.

```bash
python main.py --no-ai
```

Non-interactive auto-accept of the API-key warning prompt (CI-friendly). Runs deterministic if no key is set.

```bash
python main.py --yes
```

Non-TTY context (piped stdin). Auto-proceeds in deterministic mode with a one-line stderr notice.

```bash
python main.py < /dev/null
```

AI mode with Anthropic Claude. `--verbose` prints each row's AI explanation inline.

```bash
ANTHROPIC_API_KEY=sk-ant-... python main.py --verbose
```

AI mode with OpenAI. Same JSON contract, different provider.

```bash
OPENAI_API_KEY=sk-... python main.py --ai-provider openai --verbose
```

ASCII-art guided menu for non-technical reviewers. Choose `[3]` for deterministic, `[Q]` to quit.

```bash
python main.py --interactive
```

Web UI — branded header, file upload, kill-switch status in sidebar, expandable AI explanation per ERROR row, CSV download button.

```bash
streamlit run app.py
```

### Multi-client configuration (bonus challenge)

Strict rules: `rate_tolerance=0`, `max_hours_enforcement=true`, `overbilling_threshold=0`. Sample data → 4 ERROR rows.

```bash
python main.py --client client_a --no-ai
```

Relaxed rules: `rate_tolerance=2`, `max_hours_enforcement=false`, `overbilling_threshold=1`. Same sample data → fewer ERROR rows.

```bash
python main.py --client client_b --no-ai
```

Onboard a new client without code changes — add a block to `config/client_rules.json`, then run with the new name.

```bash
python main.py --client <new_client_name>
```

### Governance controls

Per-agent ISO-timestamped audit trail (START / DONE / ERROR / HALT / COMPLETE).

```bash
cat data/output/audit.log
```

Three roles (analyst / admin / viewer) plus per-agent permission scope.

```bash
cat config/rbac.json
```

Halt the pipeline via the kill switch.

```bash
echo '{"active": false}' > config/kill_switch.json
python main.py --no-ai
```

Re-enable the pipeline.

```bash
echo '{"active": true}' > config/kill_switch.json
```

### Validation flags (data-driven)

The bundled sample dataset exercises four of the nine flags (`RATE_MISMATCH`, `OVERBILLING`, `CONTRACT_VIOLATION`, `BILLING_OVER_MAX`). The remaining five require synthetic data and are covered by dedicated unit tests below.

Sample-data run including `Capped_Expected_Amount` and `Over_Cap_Exposure` columns.

```bash
python main.py --no-ai && cat data/output/validation_report.csv
```

`UNDERBILLING` — billed hours below worked hours (revenue leakage).

```bash
pytest tests/test_validation.py::TestUnderbilling -v
```

`BILLING_OVER_MAX` independent of `CONTRACT_VIOLATION` — invoice exceeds cap even when worked is within cap.

```bash
pytest tests/test_validation.py::TestBillingOverMaxIndependentOfWorked -v
```

`GHOST_BILLING` — billing row with no matching timesheet.

```bash
pytest tests/test_validation.py::TestGhostBilling -v
```

`MISSING_BILLING` — timesheet row with no corresponding billing (unbilled work).

```bash
pytest tests/test_validation.py::TestMissingBilling -v
```

`MISSING_CONTRACT` — project not present in `contracts.csv`.

```bash
pytest tests/test_validation.py::TestMissingContract -v
```

`DUPLICATE_RECORD` — two billing rows for the same Employee + Project.

```bash
pytest tests/test_validation.py::TestDuplicateRecord -v
```

### AI explanation contract

Inspect one ERROR row's full schema-versioned JSON contract — `schema_version`, `record`, `flags`, `explanation`, `corrective_action`, `financial_deviation` (with `capped_expected_amount` and `over_cap_exposure`), `human_review`, `confidence`, `risk_score`, `metadata`.

```bash
python main.py --no-ai
python -c "import pandas, json; df = pandas.read_csv('data/output/validation_report.csv'); print(json.dumps(json.loads(df[df.Status=='ERROR'].iloc[0].AI_Explanation), indent=2))"
```

All contract validation paths: schema enforcement, deterministic fallback on parse error, fallback on missing fields, risk-score bands.

```bash
pytest tests/test_validation.py::TestAIExplanationContract -v
```

API-key force-validation flow: `--no-ai` skips, non-TTY auto-accepts, `--yes` accepts, key-present skips, openai-provider checks the right env var.

```bash
pytest tests/test_cli.py -v
```
